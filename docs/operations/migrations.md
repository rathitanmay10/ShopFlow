# Migrations (Alembic)

## Workflow

```bash
# 1. Change a model in app/models/
$EDITOR app/models/order.py

# 2. Autogenerate a revision
uv run alembic revision --autogenerate -m "describe change"

# 3. Inspect — autogenerate misses:
#    - enum changes (new values, renames)
#    - server defaults
#    - some FK options (ondelete, ondelete cascade direction)
#    - column type narrowing (e.g. String(255) → String(64))
$EDITOR alembic/versions/000X_*.py

# 4. Apply
uv run alembic upgrade head

# 5. Verify against tests
uv run pytest
```

## Async engine in Alembic

`alembic/env.py` uses the async engine via `connection.run_sync(...)` — Alembic itself is sync, so it bridges into the same `asyncpg` connection used by the app. No separate sync DSN needed.

## Migrations on the tree

`0001..0007` (and counting). Sequential numbered files; do **not** branch revision graphs unless you know how to merge them.

## Rules

- **No business logic in migrations.** Data backfills go in a separate one-shot script under `scripts/` or a service method, invoked manually. Migrations are for schema only.
- **New enum value?** Autogenerate misses it. Hand-edit:

  ```python
  # Add a value to an existing Postgres enum
  op.execute("ALTER TYPE order_status ADD VALUE IF NOT EXISTS 'returned'")
  ```

  Note: Postgres enum values **cannot** be removed or reordered transactionally. Treat enum membership as forward-only.

- **New enum column?** Set `values_callable=lambda e: [m.value for m in e]` on the SQLAlchemy `Enum(...)` so it serializes the value, not the name. See [Conventions](../conventions.md).

- **Renames?** Autogenerate sees them as drop + add. Convert to `op.alter_column(..., new_column_name=...)` by hand to preserve data.

## Production migrate

In `docker-compose.yml` there's a `migrate` one-shot service that runs `alembic upgrade head` and exits. The `api` and `worker` services depend on it.

For deployment outside docker-compose, run:

```bash
uv run alembic upgrade head
```

before starting the API. Idempotent — safe to run on every deploy.

## Rollback

```bash
uv run alembic downgrade -1   # one step back
uv run alembic downgrade base # all the way down (destructive)
```

Downgrade scripts are autogenerated but rarely audited — verify before running in prod. Often the safer rollback is a new "fix-forward" migration.
